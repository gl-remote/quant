# 策略研发调整指引

> 目的：给 AI/开发者调整策略时做方向校准。它是指引，不是硬性限制；核心是先判断问题类型，再选择最小有效改动，避免盲目调参或堆规则。

## 1. 先定义目标形态

改策略前，先用几句话写清这次想要什么：

- 目标交易频率：例如单品种每周几次、每月几次；
- 目标平均持仓：例如数小时、1~2 天、数周；
- 样本范围：品种、时间窗口、手续费、滑点；
- 判断标准：交易次数、收益、回撤、成本占比、跨品种一致性；
- 暂停条件：什么结果说明该方向暂时不值得继续。

目标形态没定清时，参数搜索通常没有意义。

## 2. 常见现象与优先方向

| 现象 | 优先判断 | 建议方向 |
|------|----------|----------|
| 每个品种只有 0~1 笔交易 | 样本不可判定 | 先拆入场/出场结构，确认是信号稀疏还是不平仓 |
| 看似收益为正但交易极少 | 单笔持仓退化解 | 加最小交易次数约束，记录为不可用样本 |
| 交易突然很多且亏损严重 | 信号或退出过敏 | 降仓位、检查反向退出、加冷却或确认延迟 |
| 某个品种合适，其他品种很少 | 品种迁移不稳 | 记录分化，检查周期/指标是否过拟合单品种 |
| 持仓太短 | 周期或退出过敏 | 上移方向周期、加最短持仓、反向确认 N 根 |
| 持仓太长且不平仓 | 出场机制不足 | 加反向退出、时间退出或移动止盈 |
| 成本吞噬收益 | 频率过高或单笔收益太小 | 降低交易频率、提高触发质量、做成本敏感性 |
| 参数稍变结果断崖 | 没有参数平台 | 记录为不稳，考虑换结构或扩大样本 |
| 指标历史读不到或 NaN 多 | 框架数据问题 | 记录到 roadmap，避免长期在策略层缓存绕法 |

## 3. 调整顺序建议

### 3.1 先确认工具链和统计口径

收益比较应建立在这些前提上：

- 交易次数足够；
- 持仓闭环完整；
- 成本已计入；
- 指标可用；
- 出场原因语义清楚。

如果这里有问题，优先修工具链或记录框架缺陷。

### 3.2 再做信号消融

多条件 AND 的策略，优先拆开看每组信号的贡献：

- 主趋势；
- 主趋势 + 动量确认；
- 主趋势 + 回调确认；
- 全条件组合。

消融重点不是找最高收益，而是判断哪个条件让交易消失、哪个条件导致过度交易、哪些条件逻辑冲突。

### 3.3 再调整周期结构

如果信号过敏，通常先调整周期结构，再调止盈止损。

常见结构：

- 慢周期决定背景方向；
- 中周期决定入场触发；
- 快周期只用于执行或风控，不主导方向。

慢周期能降噪，也会压低频率；不要把慢周期当成频率修复器。

### 3.4 再加轻量约束

当已经有完整交易样本，但频率或持仓不舒服时，再考虑：

- 反向信号连续 N 根确认；
- 平仓后冷却；
- 最短持仓；
- 最长持仓或时间退出；
- 移动止盈。

这些约束通常改善交易质量，但不能创造 alpha。基础信号稀疏时，继续堆约束价值有限。

### 3.5 最后做参数搜索

只有样本形态接近目标时，再扩大 trial 或做参数搜索。

参数搜索前最好确认：

- 每个样本段交易次数足够；
- 没有明显单笔收益主导；
- 附近参数不是断崖；
- 成本敏感性不过分脆弱。

## 4. 实现建议

优先用现有切面/DSL 表达策略意图，避免在策略类中重复实现方向、确认、止盈止损、冷却等通用逻辑。实现新策略或调整策略前，先看 `strategy_aspects` 的模块说明。

- `data_requirements()` 只补充策略特有数据需求；DSL 涉及的指标会自动合并；
- `@trend_long` / `@trend_short` 声明趋势方向；
- `@confirm_long` / `@confirm_short` 声明确认条件；
- `@exit_for_take_profit` / `@exit_for_stop_loss` 声明出场风控建议；
- `@entry_block_*` 声明入场阻断建议。

DSL 切面只写入 `ctx.aspects`，不直接交易；交易决策仍由策略 `on_bar()` 消费这些建议完成。

如果只是验证结构假设或满足测试，可以先用策略内常量，确认有效后再参数化。需要用户配置、优化器搜索或复现实验的参数，再放入策略参数类和配置文件。

## 5. 参数搜索与持久化

策略参数可以是数值，也可以是类别。

- 数值参数：用于连续/离散搜索和数值分析；
- 类别参数：用于 profile、模式、实验标签，搜索时用 `categorical`，不建议手动映射成数字；
- 持久化时，`param_value` 保存数值，`param_type` / `param_text` 保存原始类型和值。

## 6. 框架问题处理

策略研发中遇到框架问题，建议区分“临时绕法”和“长期修复”。

适合记录到 `docs/roadmap/` 的问题包括：

- 指标历史读取不可靠；
- 指标 warmup 语义不清；
- 出场 DSL 语义不清；
- 回测持久化丢失策略配置或不能复现实验参数；
- 撮合、平仓配对、爆仓统计出现可复现异常；
- 报告缺少判断策略成色的关键指标。

临时绕法可以存在，但最好写清为什么不是最终方案，以及什么时候可以删除。

## 7. 实验记录最小信息

每轮实验尽量记录：

- 策略版本或 commit；
- 回测命令和配置；
- 品种、窗口、手续费、滑点；
- profile / 关键参数；
- 交易次数、平均持仓、收益、回撤、成本；
- 与上一轮相比改善了什么；
- 是否进入下一轮，若否，原因是什么。

如果一轮实验不能回答明确问题，通常不值得跑。
